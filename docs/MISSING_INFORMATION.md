# Недостающая информация для реализации

Ниже перечислены вопросы, без которых реализация будет опираться на предположения и может потребовать переработки.

## 1) Бизнес-правила и продуктовые решения
1. Какие конкретно пары рынков считаются целевыми для спреда?
   - Только `spot vs perpetual futures`? да 
   - Нужны ли quarterly futures? нет 
2. Какие символы включать/исключать (stablecoins, low-liquidity, leverage tokens)? всё 
3. Нужна ли фильтрация по минимальному объему/ликвидности?нет
4. Как трактовать отрицательный spread в downstream-потребителях? спред строится так что спот всегда меньше фьючерс 
5. Какой целевой формат потребления snapshot: только файл или ещё API/stream? файл 

## 2) Параметры SLA/SLO
1. Максимально допустимая задержка snapshot (end-to-end latency). RTT не учитываем, а так менее 300 мс и спайки до 1000мс 
2. Допустимый процент stale-значений в единицу времени. пока не рассматриваем
3. Целевая доступность сервиса (например, 99.5%/99.9%). 99%
4. Допустимое время восстановления после сбоя (RTO/RPO). не критично 

## 3) Технические настройки по умолчанию
1. Значение `STALENESS_THRESHOLD` (мс/сек). 30 секунд
2. Частота генерации snapshot (например, 1с / 5с / 10с). снапшот делается один раз, и на направление и конкретный символ ставится ограничение - не присылать в течение часа 
3. Размеры SHM: мы же делаем подписки на вебсокет по списку и вот исходя из этих списков мы строит shm
   - `MAX_SLOTS`
   - `SLOT_SIZE`
   - оценка верхней границы по символам.
4. Политика TTL и ротации snapshot-файлов. ротация не требуется ттл для снапшотов отсуствует 
5. Политика ротации логов и лимиты на диск. нету
