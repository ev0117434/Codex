# Недостающая информация для реализации

Ниже перечислены вопросы, без которых реализация будет опираться на предположения и может потребовать переработки.

## 1) Бизнес-правила и продуктовые решения
1. Какие конкретно пары рынков считаются целевыми для спреда?
   - Только `spot vs perpetual futures`?
   - Нужны ли quarterly futures?
2. Какие символы включать/исключать (stablecoins, low-liquidity, leverage tokens)?
3. Нужна ли фильтрация по минимальному объему/ликвидности?
4. Как трактовать отрицательный spread в downstream-потребителях?
5. Какой целевой формат потребления snapshot: только файл или ещё API/stream?

## 2) Параметры SLA/SLO
1. Максимально допустимая задержка snapshot (end-to-end latency).
2. Допустимый процент stale-значений в единицу времени.
3. Целевая доступность сервиса (например, 99.5%/99.9%).
4. Допустимое время восстановления после сбоя (RTO/RPO).

## 3) Технические настройки по умолчанию
1. Значение `STALENESS_THRESHOLD` (мс/сек).
2. Частота генерации snapshot (например, 1с / 5с / 10с).
3. Размеры SHM:
   - `MAX_SLOTS`
   - `SLOT_SIZE`
   - оценка верхней границы по символам.
4. Политика TTL и ротации snapshot-файлов.
5. Политика ротации логов и лимиты на диск.

## 4) Источники данных и устойчивость
1. Подтвержденные production endpoints (REST/WS) и их версионирование.
2. Лимиты API/WS для аккаунта/IP и стратегия при rate limit.
3. Нужны ли резервные источники при недоступности биржи?
4. Нужна ли прокси/региональная маршрутизация?

## 5) Контракты данных
1. Финальная схема `Quote` (типы, nullable-поля, единицы времени).
2. Канонические названия рынков/бирж (строгий enum).
3. Поле-источник времени как «истина»: `ts_exchange` vs `ts_receive`.
4. Версионирование формата snapshot и backward compatibility.

## 6) Операционная модель
1. Где будет запуск: single host / VM / container / orchestrator?
2. Требования по observability-стеку (Prometheus, Loki, ELK и т.д.).
3. Каналы и правила алертов (PagerDuty, Telegram, Slack).
4. Кто on-call и какие окна поддержки?

## 7) Безопасность и соответствие
1. Нужны ли секреты/ключи (если да — где хранить: env, vault)?
2. Требования к аудиту действий и хранению логов.
3. Ограничения по комплаенсу/юрисдикции.

## 8) Критерии приемки
1. Явный список acceptance-критериев для v1.1 (измеримые метрики).
2. Набор обязательных интеграционных тестов для релиза.
3. Кто подписывает релиз (роль/команда).

---

## Минимальный набор решений для старта (если нужно идти быстро)
Если нужно начать немедленно, рекомендуется зафиксировать временные дефолты:
- `STALENESS_THRESHOLD = 3s`
- `snapshot_period = 1s`
- `spot vs perpetual futures`
- исключение неактивных и leverage-токенов
- SLA черновой: snapshot latency p95 < 2s, stale < 5%

Эти значения стоит оформить как **временные допущения** и пересмотреть после первой недели наблюдений.
