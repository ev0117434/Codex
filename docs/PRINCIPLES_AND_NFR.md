# Principles and Non-Functional Requirements

## Архитектурные принципы

### P1. Простота важнее преждевременной оптимизации
Берём минимально достаточные решения, чтобы быстрее стабилизировать прод-поток.

### P2. Чёткие границы ответственности
Каждый модуль делает одну функцию (discovery, collection, normalize, compute).

### P3. Наблюдаемость по умолчанию
Логи и метрики не «добавляются потом», а встраиваются сразу.

### P4. Детерминированная деградация
При stale/потерях каналов система не падает молча: пишет `N/A`, логирует причину.

### P5. Эволюционность
Решения v1.1 должны безболезненно расширяться в v1.2+.

## NFR для v1.1

### Надёжность
- Автовосстановление WS-подключений.
- Безопасная инициализация SHM по MAGIC/VERSION.
- Атомарная публикация snapshot (tmp + rename).

### Производительность
- Низкая межпроцессная задержка за счёт SHM.
- O(1) доступ к последней котировке через индексированный слот.

### Корректность
- Валидация входных данных.
- Проверка stale-данных перед расчётом.
- Консистентное чтение слота через seqlock.

### Эксплуатация
- Health checks и диагностируемые ошибки.
- Минимальный runbook для on-call.

### Поддерживаемость
- Документированные контракты модулей.
- ADR для всех критичных выборов.
