# Architecture Overview (v1.1)

## Контекст
Система получает рыночные котировки (best bid/ask) с двух бирж и двух рынков (spot/futures), нормализует их, сохраняет в общую память и рассчитывает спреды для snapshot-отчёта.

## High-level pipeline
`symbol_discovery -> collectors -> normalizer -> shm -> spread_reader -> snapshots`

## Компоненты

### 1. symbol_discovery
**Назначение:**
- Получить активные торговые пары из Binance/Bybit (spot/futures).
- Нормализовать символы в единый вид.
- Построить пересечения и сформировать списки подписок.

**Выход:** `subscription_lists.yaml`

### 2. collectors
**Назначение:**
- Подписка на WS best bid/ask по 4 источникам.
- Реконнекты, heartbeat, backoff.
- Передача сырых сообщений в внутреннюю очередь.

### 3. normalizer
**Назначение:**
- Преобразовать 4 формата бирж в единый `Quote`.
- Провалидировать значения и timestamp.
- Подготовить запись в SHM.

### 4. shm
**Назначение:**
- Быстрый межпроцессный обмен котировками.
- Хранение последнего состояния по каждому ключу (`exchange + market + symbol`).
- Консистентное чтение/запись через seqlock.

### 5. spread_reader
**Назначение:**
- Считать свежие пары котировок из SHM.
- Рассчитать spread по формуле.
- Сгенерировать snapshot-файл атомарной записью.

## Почему это оптимально для v1.1
1. Разделение на процессы упрощает локализацию ошибок.
2. SHM уменьшает накладные расходы между этапами.
3. Snapshot-текст даёт быстрый и прозрачный артефакт для интеграции.
4. Архитектура расширяется без ломки контуров (новые биржи/рынки — через адаптеры).

## Ограничения v1.1
- Без долговременного хранилища истории.
- Без UI/дашборда.
- Без распределённого деплоя (single-host first).
