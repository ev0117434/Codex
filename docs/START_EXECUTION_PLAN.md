# Стартовый план: что нужно сделать, чтобы начать выполнение

Этот документ — единая точка входа перед началом реализации v1.1.

## 1) Цель старта
Перейти от «архитектура на бумаге» к минимально рабочему каркасу проекта, готовому для реализации T1–T5.

## 2) Definition of Ready (обязательные условия до активной разработки)

### 2.1 Закрыть оставшиеся решения по требованиям
Нужно утвердить:
1. DATA-01: production REST/WS endpoints для Binance/Bybit (spot/perp).
2. DATA-02: лимиты API/WS и стратегия rate-limit/reconnect.
3. TECH-03: размеры SHM (`MAX_SLOTS`, `SLOT_SIZE`).
4. SCHEMA-01/02/04: финальная `Quote`-схема, enum значений, версия snapshot.
5. OPS-02/04: observability baseline и операционная модель поддержки.
6. ACC-01/02/03: критерии готовности v1.1, обязательные интеграционные тесты, финальный approver.

**Выходной артефакт:** обновлённый `docs/MISSING_INFORMATION.md` без `pending` по критическим пунктам.

### 2.2 Зафиксировать стартовые технические артефакты (T0)
Создать:
- структуру директорий: `symbol_discovery`, `collectors`, `normalizer`, `shm`, `spread_reader`, `snapshot`;
- `requirements.txt`;
- `Makefile` с командами `init`, `lint`, `test`, `run`;
- `config/config.yaml` с параметрами по умолчанию;
- базовый logging + health-check + каркас метрик.

**Выходной артефакт:** локальный запуск всех процессов в «пустом» режиме без падений.

## 3) Порядок выполнения (критический путь)
1. **T0** — каркас проекта и базовая инфраструктура.
2. **T1** — discovery инструментов и генерация `subscription_lists.yaml`.
3. **T2** — 4 WS-коллектора с reconnect/backoff/heartbeat.
4. **T3** — нормализация raw payload в `Quote` + валидации.
5. **T4** — POSIX SHM (layout, writer/reader, seqlock).
6. **T5** — spread-логика + атомарный snapshot + cooldown.
7. **T7 (P0)** — минимальный контур unit/integration тестов.
8. **T6/T8** — наблюдаемость и эксплуатационная готовность.

## 4) Минимум перед первой строкой «боевого» кода
- Утверждены open decisions из раздела 2.1.
- Созданы артефакты T0 из раздела 2.2.
- Есть один согласованный пример end-to-end сценария для 1 символа на 4 источниках.

## 5) Критерий «можно начинать реализацию»
Можно переходить к T1/T2, когда одновременно выполнено:
1. Нет критических `pending/delegated` в решениях по требованиям.
2. T0 полностью поднят локально.
3. Прогон `make test` (или эквивалент) проходит на каркасе без падений.

## 6) Что делать первым коммитом
1. `chore: bootstrap project skeleton for v1.1`
2. Содержимое коммита:
   - базовая структура модулей;
   - конфиг, Makefile, requirements;
   - базовые точки входа процессов;
   - health-check и логирование.

Это создаст стабильную основу, после которой реализация T1–T5 пойдёт без лишних возвратов к структуре.
